## 统一美术方向
所有瓦片资源统一遵循以下风格：

- 主题：中世纪暗黑城镇
- 视角：正交俯视，适配 Godot 2D TileMap
- 视觉关键词：磨损、尘土、旧石材、枯草、泥痕、低饱和
- 配色倾向：炭灰、土褐、脏米色、暗苔绿
- 光照要求：平面游戏可用，无明显投射阴影

## 基础尺寸规则
当前地图资源主规格如下：

- 单瓦片尺寸：`64x64`
- 常用图集规格：
- `5x5` 图集，总尺寸 `320x320`
- `10x5` 图集，总尺寸 `640x320`
- 资源默认服务于 `1920x1080` 视口下的较大瓦片显示方案

后续如果继续生产同系列瓦片，优先保持 `64x64`，避免混入 `16x16` 或 `32x32` 的旧试验规格。

## 图层职责
当前地图资源建议按三层理解：

- `ground`：底层地表，负责大面积铺地
- `road`：道路层，负责自动连路或手动路网
- `decoration`：装饰层，负责污渍、碎屑、草痕、小型附着元素

图层职责原则：

- `ground` 必须能独立大面积平铺
- `road` 必须和 `ground` 边缘自然融合
- `decoration` 必须是可叠加层，不应像独立道具块一样悬浮

## 地面资源规则
地面图集的生产规则如下：

- 使用单一大面积地表纹理做来源
- 再本地裁切为 `5x5` 个 `64x64` 瓦片
- 每个瓦片边缘做基础连续性处理
- 整体偏暗，细节丰富，但不能有强方向性阴影

地面内容以以下材质混合为主：

- 旧石板
- 脏土
- 泥地
- 少量苔藓
- 枯草痕迹
- 细碎砾石

## 路面资源规则
路面资源不是独立底板，而是“亮一些的路面材质叠加在暗地面上”。

核心原则：

- 路面比地面亮一点，但不能跳出整体暗黑风格
- 边缘必须向 `ground` 柔和过渡
- 不要做硬边矩形道路
- 路面连接方向必须稳定

当前路面总图集 [road_tileset_64x64_10x5_combined.png](/C:/Users/cc/data/godot/tower/resource/image/road_tileset_64x64_10x5_combined.png) 的逻辑分布：

- 左半 `5x5`：直路和拐角
- 右半 `5x5`：T 字路口和十字路口

路型语义如下：

- `pass_vertical`：上下通行
- `pass_horizontal`：左右通行
- `corner_up_left`：左上拐角
- `corner_up_right`：右上拐角
- `corner_down_left`：左下拐角
- `corner_down_right`：右下拐角
- `t_open_top`：缺上口 T 字
- `t_open_right`：缺右口 T 字
- `t_open_bottom`：缺下口 T 字
- `t_open_left`：缺左口 T 字
- `cross`：十字路口

## 装饰资源规则
装饰资源分两类思路：

- 老方案：块状装饰，适合独立点缀，但容易悬浮
- 新方案：附着型装饰，适合叠在 `ground` 上

本项目后续优先采用“附着型装饰”。

附着型装饰规则：

- 装饰应更薄、更散、更贴地
- 轮廓不要像单独道具块
- 要保留大量透明区域
- 边缘应破碎、柔和、不规则
- 元素密度不能太满，否则会抢地面层信息

推荐内容：

- 苔藓痕迹
- 枯草痕迹
- 细碎砾石
- 灰烬污渍
- 脏泥拖痕
- 裂纹和磨损
- 砖石碎片
- 稻草碎屑
- 轻微车辙
- 脚印或磨耗痕迹

不推荐内容：

- 大块独立石堆
- 完整箱子木桶
- 强烈抬起感的杂物
- 轮廓很完整的道具块

## 透明边处理经验
此前最大的坑是 chroma key 绿色底在透明边缘残留荧光。

经验总结如下：

- 仅仅把纯绿色像素设为透明是不够的
- 半透明边缘像素的 RGB 也必须去绿
- 完全透明像素的 RGB 不能保留绿色，否则引擎滤波时仍会采样到绿边
- 最稳的处理方式是：
- 将半透明边缘像素颜色替换为邻近装饰本体颜色
- 将完全透明像素颜色也改成邻近土色，但 alpha 仍为 `0`

## Godot 导入建议
对于所有带透明边缘的装饰图集，Godot 导入时建议：

- 关闭 `Filter`
- 关闭 `Mipmaps`
- 修改后重新 `Reimport`

原因：

- 开启过滤会把透明区 RGB 采样出来
- 开启 mipmaps 会进一步放大边缘污染
- 即使图本身已经去绿，错误导入参数也可能重新引出边缘发光

## 原生透明方案规则
如果后续要继续做更大件的资源，比如建筑、树木、岩石，建议优先走“原生透明”而不是绿底抠图。

原因：

- 建筑树木岩石这类元素体积更大
- 边缘复杂度更高
- 抠图残留更明显
- 直接透明 PNG 更适合作为独立装饰或障碍物图集

原生透明方案要求：

- 使用 CLI 路径
- 使用 `gpt-image-1.5`
- 使用 `--background transparent`
- 输出 `png`
- 依赖本机配置 `OPENAI_API_KEY`

注意：

- 当前会话中 CLI 尚未真正执行，因为环境变量在当前终端里未读到
- 新会话如果继续这条路线，先检查 `OPENAI_API_KEY` 是否在新终端中可见

## 推荐的后续大件图集规格
如果下一会话继续做建筑、树木、岩石图集，建议采用：

- 单格尺寸：`128x128`
- 布局：`5x5`
- 总尺寸：`640x640`
- 背景：原生透明

建议内容：

- 小型石屋
- 破旧木屋
- 枯树
- 歪斜老树
- 岩石堆
- 裂石
- 小废墟墙体
- 木桶木箱组合
- 火盆或路灯
- 井口或石基

## 新会话接手建议
新会话接手时建议按以下顺序继续：

1. 先阅读本文件
2. 检查现有图集和 TileSet 是否仍在当前路径
3. 如果继续做装饰层，优先以 `ground_decor_tileset_64x64_5x5_v2.png` 为参考风格
4. 如果继续做路网，不要重做 `road.tres`，优先复用当前 terrain 和 custom data 结构
5. 如果继续做原生透明大件图集，先确认 `OPENAI_API_KEY` 在当前终端可读
6. 生成前明确：
- 图集布局
- 单格尺寸
- 是否地表附着型
- 是否独立物件型
- 是否需要直接透明输出

## 不要重复踩的坑
- 不要把 `16x16` 老规格和当前 `64x64` 资源混用
- 不要把装饰层做成厚重独立块
- 不要只删纯绿色像素就结束去底
- 不要忽略透明像素的 RGB 底色
- 不要在未确认环境变量的情况下直接走 CLI 原生透明方案
- 不要在新会话里重新猜路面图集语义，直接复用现有命名和分布

当前项目已经形成了一套可继续扩展的地图资源生产基线：

- `ground` 负责底层铺地

- 所有资源统一使用中世纪暗黑城镇风格
