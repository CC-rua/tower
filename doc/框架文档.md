# 框架文档

> 本文档用于定义塔防项目的技术框架、目录结构、系统边界与开发约定。

## 1. 目标

在不进入具体业务开发的前提下，先搭建一套可持续扩展的 Godot 项目框架，沉淀基础设施能力，并对已导入插件进行统一封装，降低后续业务开发的耦合度。

## 2. 技术选型

- 引擎：Godot 4
- 语言：GDScript
- 版本管理：Git
- 已接入插件：Dialogue Manager、GDExcelExporter、Scene Manager、Phantom Camera

## 3. 框架设计原则

- 基础设施先行，业务系统后置。
- 对第三方插件做项目内封装，避免业务层直接依赖插件 API。
- 模块职责清晰，便于后续扩展塔、防御、敌人、波次和地图系统。
- 场景、配置、存档等核心能力保持统一入口。
- 自动加载单例数量保持克制，只保留明确具备全局职责的系统入口。

## 4. 目录结构规划

当前目录规划将 `systems` 作为无场景依赖的基础框架入口，场景及其挂载脚本优先聚合在 `scenes` 对应目录下：

- `systems/scene_flow`：场景切换与流程编排。
- `systems/save`：存档读写、存档结构与版本兼容。
- `systems/config`：配置加载、缓存与查询入口。
- `systems/dialogue`：剧情对话能力封装。
- `scenes/app`：主场景与应用级容器，主场景挂载脚本与场景同目录。
- `scenes/pages`：启动页、主菜单、关卡选择、战斗、结算等页面场景及其挂载脚本。
- `scenes/battle/levels`：战斗关卡场景、地图控制器、路线控制器等关卡级挂载脚本。
- `scenes/battle/objects`：塔位、宝石、路线标记、障碍等战斗对象场景及其挂载脚本。
- `scenes/battle/enemy`：敌人场景及其挂载脚本。
- `scenes/battle/ui`：战斗内复用 UI 子场景及其挂载脚本。
- `data/excel`：Excel 原始配置文件存放目录。
- `data/config_gdscript`：由插件导出的 GDScript 配置文件目录。
- `autoload`：自动加载脚本入口。
- `resources/save`：存档结构资源或默认模板。

## 5. 核心模块划分

当前已明确优先建设以下基础模块：

- 场景切换模块
- 存档模块
- 配置读取模块
- 插件封装层

## 6. 关键系统边界

### 6.1 场景流程系统

- 项目采用“单一主场景挂载子场景”的组织方式。
- 主场景作为应用运行容器，负责挂载当前页面场景、全局 UI 层以及转场表现层。
- 启动页、主菜单、关卡选择、战斗场景、结算场景作为子场景，由统一的场景流程模块负责切换。
- 页面级切换默认通过主场景挂载子场景完成，不依赖整场景重载。
- `SceneManager` 插件仅用于整场景切换，例如从主页面进入完整游戏流程、加载存档进入运行态等需要全局转场的场合。
- `SceneManager` 插件能力不直接暴露给业务层，由项目内 `scene_flow` 模块统一封装。
- 场景切换应支持进入前检查、切换锁、切换动画、切换完成通知等基础能力。

### 6.2 存档系统

- 项目首期采用单存档文件方案。
- 单存档中统一保存设置项、玩家进度、已解锁内容、关卡星级与局外成长数据。
- 存档模块负责读写、默认值填充、版本号记录与兼容升级入口。
- 建议将“存档数据结构定义”与“存档文件读写逻辑”分开，避免数据结构和文件操作耦合。

### 6.3 配置系统

- 配置来源为 Excel，由编辑器阶段导出为 GDScript。
- 运行时只读取导出结果，不承担热重载职责。
- `data/excel` 用于存放原始表格。
- `data/config_gdscript` 用于存放导出的 GDScript 配置数据。
- 插件生成的 `settings.gd` 作为原始配置总入口，由 `ConfigManager` 在初始化阶段统一加载，而不再额外挂载为独立全局单例。
- 业务层不应直接依赖导出脚本路径，而应通过统一配置入口查询数据。
- 建议配置系统负责首轮加载与缓存，业务层只关心查询接口与结果格式。

### 6.4 对话系统

- 对话系统主要用于剧情对话处理。
- 对话系统与场景流程允许联动，但尽量依赖插件已有能力与项目封装接口完成，不在首期重复实现底层播放机制。
- 项目侧主要负责定义“何时触发对话、对话结束后如何继续流程”的接入规范。

## 7. 数据与配置约定

- Excel 文件统一存放于 `data/excel`。
- Excel 导出的 GDScript 配置统一存放于 `data/config_gdscript`。
- 运行时配置读取只面向导出产物，不直接解析 Excel。
- 每张导出表在业务侧不直接使用裸字典行数据，而是通过对应的 `XxxRef` 对象承载字段，并在对象内完成类型约束。
- 建议采用“一张表对应一个 Ref 类”的约定，例如 `demo.gd` 对应 `demo_ref.gd`。
- 通用字段转换逻辑应优先沉淀到配置行父类中，避免在各个 `XxxRef` 中重复实现数组、字典、Callable 等转换方法。
- `XxxRef` 可以按业务需要提供静态访问入口，例如 `get_by_id()`、`get_all()`，其底层数据统一来自 `ConfigManager` 缓存。
- 配置系统对外提供统一查询接口，例如按表名、主键或预定义方法读取数据。
- 若后续配置量增大，应优先在 `ConfigManager` 内做缓存与索引，而不是让业务层自行缓存。
- 为兼容历史导出产物，配置系统可临时读取旧目录 `data/ref`，但新导出目标应统一为 `data/config_gdscript`。

## 8. 场景与节点组织约定

- 建议创建一个应用主场景作为唯一根入口，例如 `scenes/app/app.tscn`。
- 主场景内部至少包含页面挂载层、全局 UI 层、转场层。
- 各页面场景保持独立，不在页面内部直接控制其它页面的加载与卸载。
- 页面跳转统一通过 `SceneFlow` 发起，页面自身不直接操作插件。

## 9. 脚本编写约定

- 插件调用优先集中在对应 `systems` 模块内，避免散落到页面与业务脚本中。
- 自动加载脚本只承担全局协调职责，不承载复杂业务逻辑实现细节。
- 直接挂载到 `.tscn` 节点上的脚本，应与该场景或同级子场景放在同一目录下，便于从场景反查行为实现。
- 只有不依赖任何具体场景才能独立运行的服务、单例实现和基础设施脚本，才放入 `systems`。
- 脚本命名与目录保持一致，优先使用清晰职责命名。
- 推荐一个模块一个入口脚本，一个或多个辅助脚本，避免单文件膨胀。
- 所有方法必须添加方法级注释，说明方法用途、职责或调用时机。
- 函数入参统一使用下划线 `_` 前缀命名，例如 `_page_id`、`_config_name`。
- 变量需要配套中文注释，说明变量用途，避免仅靠变量名猜测含义。
- 当一个类中存在多个方法时，方法顺序统一为：继承的方法、信号处理方法、本类自定义方法。

## 10. 后续扩展预留

后续将为战斗、地图、塔、敌人、波次、UI 和数据驱动内容预留接入点。
后续业务模块应依赖 `SceneFlow`、`SaveManager`、`ConfigManager` 等统一入口，而不是绕开系统层直接访问底层资源。

## 11. 自动加载单例规划

首期确认以下自动加载单例：

- `App`
- `SceneFlow`
- `SaveManager`
- `ConfigManager`

推荐脚本位置：

- `autoload/app.gd`
- `autoload/scene_flow.gd`
- `autoload/save_manager.gd`
- `autoload/config_manager.gd`

职责划分建议如下：

### 11.1 App

- 作为应用级总入口，负责全局初始化顺序。
- 在启动时组织配置系统、存档系统、场景流程系统的初始化。
- 持有少量全局运行状态，例如是否完成初始化、当前运行模式等。
- 不直接处理具体页面跳转细节，不承担配置查询和存档细节逻辑。

### 11.2 SceneFlow

- 作为页面与流程的统一调度入口。
- 负责主场景下子场景的加载、替换、卸载与转场控制。
- 封装 `SceneManager` 插件能力，并对外暴露稳定接口。
- 管理启动页、主菜单、关卡选择、战斗场景、结算场景之间的跳转。

### 11.3 SaveManager

- 负责单存档文件的创建、读取、写入与版本兼容。
- 维护设置项、玩家进度、已解锁内容、关卡星级、局外成长的统一存取入口。
- 对外提供保存、读取、重置、迁移等接口。

### 11.4 ConfigManager

- 负责加载 `data/config_gdscript` 下的导出配置。
- 建立运行时缓存与统一查询入口。
- 屏蔽导出结果的底层文件结构，避免业务层直接依赖导出脚本路径。
