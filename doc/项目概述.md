# 项目概述

> 本文档用于沉淀塔防项目的产品定位、核心玩法、目标范围与阶段目标。

## 1. 项目基础信息

- 项目名称：tower（项目代号阶段）
- 项目代号：tower
- 项目类型：传统路径塔防游戏
- 目标平台：PC
- 技术方案：Godot 4 + GDScript

## 2. 项目定位

### 2.1 一句话描述

一款以传统路径防守为核心，并强调防御塔向上合成成长的奇幻风格塔防游戏。

### 2.2 目标体验

以稳定清晰的建塔防守节奏为基础，提供通过塔位经营、资源分配和塔体合成带来的中长期决策乐趣。
首期阶段不追求玩法内容堆叠，而是优先保证基础框架清晰、流程稳定、后续易扩展。

### 2.3 核心卖点

- 传统路径塔防的清晰规则与易理解战斗反馈。
- 防御塔可向上合成，强化局内成长与构筑变化。
- 以可持续扩展为前提搭建项目骨架，便于后续添加新塔、新敌人、新地图和新系统。

## 3. 核心玩法

### 3.1 基础规则

玩家通过在路径周边布置防御塔，阻止敌人沿固定路线到达终点。

### 3.2 核心循环

建塔 -> 打怪 -> 获得资源 -> 升级 -> 下一波

### 3.3 主要系统

当前规划优先建设基础框架系统，暂不进入具体业务玩法开发。首批重点支持以下能力：

- 启动页、主菜单、关卡选择、战斗场景、结算场景的基础流程串联。
- 基于单一主场景挂载子场景的场景切换框架。
- 单存档文件方案，统一承载设置项、玩家进度、解锁内容、关卡星级和局外成长。
- Excel 配置导出后在运行时只读的统一配置系统。
- 面向剧情对话的对话系统接入规范。
- 已导入插件的项目内封装，避免业务层直接依赖第三方插件接口。

## 4. 美术与表现方向

奇幻风格。

## 5. 开发范围

### 5.1 首期目标

完成可持续扩展的项目框架，优先沉淀基础设施能力，而非直接开发塔、防怪、关卡等业务内容。
框架完成后，应具备清晰的系统边界、统一的场景流程入口、可落地的存档方案与配置读取链路。

### 5.2 暂不纳入范围

- 暂不进入具体战斗业务开发。
- 暂不实现塔、防御、敌人、关卡数值等正式内容。
- 暂不处理相机系统业务接入，仅保留插件能力预留。

## 6. 风险与约束

- 项目仍处于前期准备阶段，玩法细节、系统边界和资源组织方式尚未完全确定。
- 已导入多个插件，若缺少统一封装，后续业务层将直接依赖第三方实现，增加维护成本。
- 当前首期目标偏基础设施建设，若中途混入业务功能开发，容易打乱框架边界。

## 7. 待展开提纲

### 7.1 产品定位

- 为什么选择传统路径塔防作为基础玩法。
- 为什么将“防御塔向上合成”作为核心特色。
- 首期不做业务内容、先做基础框架的原因。

### 7.2 玩家体验目标

- 单局体验节奏。
- 资源获取与塔升级的决策感。
- 合成成长带来的中期目标感。

### 7.3 首期建设范围

- 启动页、主菜单、关卡选择、战斗场景、结算场景的流程串联。
- 场景切换能力。
- 存档能力。
- Excel 配置导出与统一读取能力。
- 剧情对话能力。
- 自动加载单例与基础系统初始化流程。

### 7.4 后续玩法预留

- 防御塔系统。
- 敌人系统。
- 波次系统。
- 地图与路径系统。
- 局外成长与解锁扩展。
